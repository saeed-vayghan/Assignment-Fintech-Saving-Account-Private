# OWASP Top 10 — Service Security Analysis
 
 This document provides a brief analysis mapping each of the OWASP Top 10 security principles to a specific microservice within the Alborz Bank architecture.
 
 | # | OWASP Category | Analyzed Service | Potential Security Issue | Architectural Solution(s) |
 |---|---|---|---|---|
 | **1** | Broken Access Control | **Deposits BFF** (Team 2) | User A successfully modifies the API request to fetch User B's transaction history or deposit balances. | • Enforce Zero-Trust authorization.<br>• API Gateway Extracts `userId` from token.<br>• BFF strictly filters DB queries by `ownerId = token.userId`. |
 | **2** | Cryptographic Failures | **Auth API** (Team 4) | Session tokens, national IDs (PNR), or passwords leak in plaintext from database snapshots or transit interception. | • Mandate TLS 1.3 for all traffic.<br>• Utilize AWS KMS to enforce encryption-at-rest on DynamoDB.<br>• Hash passwords utilizing Argon2id; never store plaintext. |
 | **3** | Injection | **CAMT Ingestion Pipeline** (Team 3) | Malicious actors upload XML External Entity (XXE) payloads within the banking CAMT files, attempting Server-Side injection. | • Completely disable DTDs (Document Type Definitions) in XML parsers.<br>• Strictly validate the CAMT.053 schema before parsing memory. |
 | **4** | Insecure Design | **Transaction Processing Service** (Team 2) | Business logic flaws allow a user to withdraw funds before a deposit is fully cleared and settled. | • Implement pessimistic locking/idempotency.<br>• Strictly enforce state machine rules (e.g., `SETTLED` state required for withdrawal). |
 | **5** | Security Misconfiguration | **Regulatory S3 Archive** (Team 3) | The S3 bucket holding central bank reports is accidentally made public due to a misconfigured IAM policy or Terraform drift. | • Enable "Block Public Access" at the AWS Account level.<br>• Enforce AWS Config rules to auto-remediate public buckets. |
 | **6** | Vulnerable Components | **Notification Engine** (Team 4) | Outdated NPM dependencies for the AWS SES/SNS wrappers contain a known Remote Code Execution (RCE) vulnerability. | • Integrate Dependabot/Snyk in GitHub Actions.<br>• Block PR merges if Critical/High CVEs are detected in `package.json`. |
 | **7** | Auth Failures | **Registration API** (Team 1) | Attackers script a credential stuffing or OTP brute-forcing attack during the customer onboarding flow. | • Integrate AWS WAF rate-limiting for the login/registration routes.<br>• Implement exponential backoff and account locking after 5 failures. |
 | **8** | Data Integrity Failures | **IDV Webhook Handler** (Team 1) | An attacker intercepts the asynchronous webhook from Onfido/Signicat and modifies the payload status from `REJECTED` to `APPROVED`. | • Require the IDV provider to sign payloads.<br>• Validate cryptographic HMAC signatures (via `X-Signature` header) using a shared secret. |
 | **9** | Logging & Monitoring Failures | **Central Observability Pipeline** (Team 4) | The system fails to detect or alert when the Underwriting API experiences a sustained 100% failure rate, paralyzing onboarding. | • Configure CloudWatch Alarms for `5xx` error rates > 1%.<br>• Route alarms to PagerDuty/SNS to page the on-call engineer instantly. |
 | **10** | Server-Side Request Forgery (SSRF) | **Underwriting Decision Engine** (Team 1) | The engine fetches external compliance lists using URLs manipulatable by user input, allowing internal network scanning. | • Strictly whitelist allowed outbound domains (e.g., `api.complyadvantage.com`).<br>• Execute functions in a private VPC with strict Egress Security Groups. |
